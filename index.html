MVCC实现原理

Posted on 2021-06-07 In 数据库 , MySQL
Symbols count in article: 3.4k

MVCC 概述

多版本并发控制（Multiversion Concurrency Control，MVCC）是 InnoDB 实现的一种提高并发的技术。

早期的数据库系统，只有读读之间可以并发，读写，写读，写写都会阻塞。引入 MVCC 之后，只有写写之间相互阻塞，其他三种操作都可以并发，大幅度挺高了数据库的并发能力。

在内部实现中，InnoDB 通过 undo log 保存每条数据的多个版本，通过 undo log 可以找回数据历史版本提供给用户读。

在同一个事务中，用户只能看到该事务创建快照之前已经提交的修改和该事务本身做的修改。

InnoDB 通过 行级锁和 MVCC 实现提交读（Read Commited，RC）和可重复读（Repeatable Read，RR）。

MVCC 实现

MVCC 的实现基于：隐藏字段、Read View 和 Undo log。

隐藏字段

InnoDB 会为每行数据添加三个隐藏字段：

  • DB_TRX_ID：最近插入或更新这条记录的事务 ID。
  • DB_ROLL_PTR：指向回滚段（roll segment）中的记录的上一个版本（undo log）
  • DB_ROW_ID：如果没有指定主键，会将该字段作为主键。
Read View

在 RR 隔离级别下，执行第一个 select 语句时，会生成一个 ReadView，保存当前系统中活跃事务的 ID。

通过比较行记录当前版本和 undo log 中的旧版本中的 DB_TRX_ID 和事务的 ReadView 活跃事务的 ID，可以判断出哪个版本的数据对于该事务时可见的。

Read View 结构源码

重要字段：

  • low_limit_id：当前最大事务 ID 加 1，即下一个将被分配的事务 ID
  • trx_ids：Read View 创建时其他未提交的活跃事务 ID 列表
  • up_limit_id：trx_ids 中最小的事务 ID，如果 trx_ids 为空，则 up_limit_id 为 low_limit_id
  • creator_trx_id：当前事务 ID
undo log

undo log 记录的是数据行的旧版本信息，如果数据行对于当前事务不可见，可以根据 undo log 找到对于事务可见的数据版本。

InnoDB 将 undo log 分为两类：

  • insert undo log：事务 insert 新记录时产生的 undo log，只在事务回滚时需要，事务提交后就可以立即丢弃。
  • update undo log：通常用于一致性读中，只有在创建了快照的事务不再需要 update undo log 来构建数据的旧版本时，才能丢弃 update undo log。

update undo log 中包括删除操作，因为需要保证一致性读，数据的删除不是立即执行的。

InnoDB 在可以丢弃关于删除的 update undo log 时才真正删除数据。

现在有一个事务 ID 为 2 的事务要将这条记录的 Honor 修改为 fmvp，执行步骤如下：

  • 对这条记录加写锁（排他锁、独占锁）
  • 将该记录复制到 undo log
  • 修改 Honor，修改 DB_TRX_ID 为 2，让 DB_ROLL_PTR 指向 undo log 中的旧版本
  • 事务提交，释放写锁

可以看出，不同事务或者相同事务的对同一记录行的修改，会使该记录行的 undo log 成为一条链表，undo log 的链首就是最新的旧记录，链尾就是最早的旧记录。

记录版本可见性规则

ReadView 保存了事务开始时系统中活跃事务的 ID，其中最小值时 up_limit_id，最大值是 low_limit_id。通过记录的 DB_TRX_ID 和 up_limit_id、low_limit_id 的比较，可以判断该版本记录对于当前事务是否可见，具体判断流程如下：

  1. DB_TRX_ID < up_limit_id：说明生成该版本记录的事务在当前事务创建 ReadView 前已经提交，对于当前事务可见；

  2. DB_TRX_ID > low_limit_id：说明生成该版本记录的事务是在当前事务创建 ReadView 后才开始，对于当前事务不可见；

  3. up_limit_id < DB_TRX_ID < low_limit_id：此时需要判断 DB_TRX_ID 是否在 ReadView 的活跃事务 ID 列表中

    1. DB_TRX_ID 不在列表内：说明生成该版本记录的事务在当前事务创建 ReadView 前已经提交，对于当前事务可见；
    2. DB_TRX_ID 在列表内：说明生成该版本记录的事务在当前事务创建 ReadView 前已经开始，但还未提交，对于当前事务不可见；

快照读与当前读

事实上，提交读和可重复读指的都是基于 ReadView 的快照读，即通过记录的 DB_TRX_ID 判断记录的可见性。而更新数据时，需要当前读（current read）。

换句话说，当前读（current read）返回的是记录的当前版本；而快照读是通过判断返回数据可见的版本（利用 DB_ROLL_PTR 和 undo log ，可以找到对于当前事务可见的数据版本）。

前读（current read）需要加锁，下面这些语句都要进行当前读：

  • select k from t where id=1 lock in share mode;
  • select k from t where id=1 for update;
  • update
  • insert
  • delete

MVC 模式

MVC 是一种架构模式：

  • M(Model)：数据模型层，提供要展示的数据。
  • V(View)：展示数据。
  • C(Controller)：接收用户的请求，委托数据模型进行处理，将处理后的结果返回给视图，由视图负责展示。

其中一种 MVC 架构就是 JSP + servlet + JavaBean。

servlet 示例

  • 删掉 src 目录，新建一个 maven 模块，并添加 web app 支持

具体操作是：右键创建好的 maven 模块 -> Add Framework Support -> web app

  • 在 WEB-INF 目录下新建一个 jsp 文件夹，新建 hello.jsp
  • 编写 servlet，处理用户请求
  • 在 web.xml 中注册 HelloServlet
  • 配置 tomcat，并启动测试

分别测试：localhost:8080/hello?method=add 和 localhost:8080/hello?method=delete

DispatcherServlet

DispatcherServlet 是 SpringMVC 的核心，作用是将用户的请求转发给不同的处理器，然后将处理结果转给视图解析器进行处理，最后将视图解析器的处理结果返回给用户。
