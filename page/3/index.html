Git 基本理论

Git 本地有三个工作区域：

  1. 工作目录（Workspace）
  2. 暂存区（Index/Stage）
  3. 本地仓库（Repository）

加上远程仓库（Remote），一共四个区域，文件在这四个区域的转换关系如下：

1. 仓库

1.1 本地仓库

  1. 创建一个仓库：git init

    可以在一个空目录下执行该命令，也可以在一个已有项目文件夹下执行。

  2. 添加文件到暂存区：git add <file>

    此时文件被添加到暂存区（stage）

  3. 提交文件到仓库：git commit -m 'message'

    把暂存区中的内容提交到当前分支

    git add 可多次执行后统一 commit

    message 是本次提交的说明信息

1.2 远程仓库

克隆远程仓库：git clone 远程仓库地址

关联单个仓库：git remote add origin 远程仓库地址

关联多个仓库：

  1. 关联 github 仓库：git remote add github_origin 远程仓库地址

  2. 关联 gitee 仓库：git remote add gitee_origin 远程仓库地址

2. 工作区、暂存区、版本库

工作区：执行 git init 的目录

版本库：.git 目录（在工作区中）

  • 暂存区（state）
  • master 分支
  • HEAD 指针（指向当前版本）

2.1 查看工作区状态

查看哪些文件被修改过：git status

查看特定文件的修改内容：git diff <file>

2.2 版本回退

查看版本（提交）历史：git log

这是完整的输出信息，如果觉得信息太多，加上参数

git log --pretty=oneline

在 git 中，用 HEAD 表示当前版本，HEAD^ 表示上个版本，HEAD^^ 表示上上个版本，HEAD～100 表示往上 100 个版本:

  • 回到上个版本：git reset --hard HEAD^
  • 回到某个旧版本：git reset —hard commit_id

回到未来版本，分两步：

  1. git reflog（查看 commit_id）
  2. git reset --hard commit_id

3. 撤销修改

回到最近一次 git commit 或 git add时的状态：git checkout -- <file>

把暂存区的修改退回到工作区：git reset HEAD <file>

接着如果要彻底撤销修改就执行 git checkout -- <file>

4. 删除文件

工作区删除了一个文件 file，如果版本库也要删除，执行：git rm <file>

如果工作区是误删，想要恢复，执行：git checkout -- <file>

也就是回到最近一次 git commit 的状态

不过恢复的 file 是上次 commit 后的版本，如果自上次 commit 后 file 修改过，这部分修改会丢失

5. 分支管理

HEAD严格来说不是指向提交，而是指向master，master才是指向提交的，所以，HEAD指向的就是当前分支。

一开始的时候，master分支是一条线，Git用master指向最新的提交，再用HEAD指向master，就能确定当前分支，以及当前分支的提交点：

每次提交，master分支都会向前移动一步，这样，随着你不断提交，master分支的线也越来越长。

当我们创建新的分支，例如dev时，Git新建了一个指针叫dev，指向master相同的提交，再把HEAD指向dev，就表示当前分支在dev上：

Git创建一个分支很快，因为除了增加一个dev指针，改改HEAD的指向，工作区的文件都没有任何变化！

使用命令 git checkout -b dev（创建并切换到分支 dev） 或 git switch -c dev（创建并切换到分支 dev）

不过，从现在开始，对工作区的修改和提交就是针对dev分支了，比如新提交一次后，dev指针往前移动一步，而master指针不变：

假如我们在dev上的工作完成了，就可以把dev合并到master上。Git怎么合并呢？最简单的方法，就是直接把master指向dev的当前提交，就完成了合并：

命令如下：

  1. git checkout master / git switch master
  2. git merge dev（合并指定分支 dev 到当前分支 master，第一步已经切换到 master 分支）

合并完分支后，可以删除dev分支。删除dev分支就是把dev指针给删掉，删掉后，我们就剩下了一条master分支：git branch -d dev

5.1 分支管理策略

通常，合并分支时，如果可能，Git会用Fast forward模式，这种模式下，删除分支后，会丢掉分支信息。

如果要强制禁用Fast forward模式，Git就会在merge时生成一个新的commit，这样，从分支历史上就可以看出分支信息。

下面我们实战一下–no-ff方式的git merge：

创建分支

git switch -c dev

修改文件并提交

git add

git commit -m ‘分支提交’

切换回 master

git switch master

合并 dev 分支

git merge --no-ff -m ‘merge with no-ff’ dev

合并分支时，加上–no-ff参数就可以用普通模式合并，合并后的历史有分支，能看出来曾经做过合并，而fast forward合并就看不出来曾经做过合并。

6. 解决冲突

场景：

  1. 创建并切换到分支：git switch -c feature1(分支名)
  2. 修改某个文件并提交：
    1. git add 或 git add .(表示提交所有文件)
    2. git commit -m '分支 feature1 的修改'
  3. 切换回 master 分支：git switch master
  4. 修改同一文件并提交
    1. git add 或 git add .(表示提交所有文件)
    2. git commit -m ‘master 的修改’

现在 master 分支和 feature1 分支都有提交，如下图：

  1. 合并 feature1 分支：git merge feature1

这个时候会有冲突：

Auto-merging readme.txt（一个文件名）

Automatic merge failed; fix conflicts and then commit the result.

解决：

查看 file，修改成希望的那样，然后提交：

git add

git commit -m ‘修复冲突’

用带参数的 git log 可以看到分支的合并情况：git log --graph --pretty=oneline --abbrev-commit

7. Bug 分支

软件开发中，bug 就像家常便饭一样。有了 bug 就需要修复，在Git 中，由于分支是如此的强大，所以，每个 bug 都可以通过一个新的临时分支来修复，修复后，合并分支，然后将临时分支删除。

当你接到一个修复一个代号 101 的 bug 的任务时，很自然地，你想创建一个分支 issue-101 来修复它，但是当前正在 dev 上进行的工作还没有提交。Git 提供了一个stash功能，可以把当前工作现场“储藏”起来，等以后恢复现场后继续工作：

  1. 存储工作现场：git stash

  2. 切回主分支修复 bug：git switch master

  3. 创建临时分支来修复 bug：

    1. git switch -c issue-101

    2. git add

    3. git commit -m ‘fix bug 101’

      提交信息： [issue-101 4c805e2] fix bug 101

  4. 切回主分支：git switch master

  5. 合并修复分支：git merge --no-ff -m "merged bug fix 101" issue-101

  6. 删除修复分支：git branch -d issue-101

  7. 现在回到 dev 分支继续工作：git swtich dev

  8. 查看工作现场：git stash list

  9. 恢复工作现场，两种方法：

    1. git stash pop

      恢复之后再 git stash list 就看不到任何东西了

    2. git stash apply + git stash drop

在master分支上修复了bug后，我们要想一想，dev分支是早期从master分支分出来的，所以，这个bug其实在当前dev分支上也存在。

同样的bug，要在dev上修复，我们只需要把4c805e2 fix bug 101这个提交所做的修改“复制”到dev分支。

切换到 dev 分支：git switch dev

复制特定提交到当前分支：git cherry-pick 4c805e2

如果要丢弃一个没有被合并过的分支，可以通过 git branch -D <name> 强行删除。

8. 多人协作

当你从远程仓库克隆时，实际上Git自动把本地的master分支和远程的master分支对应起来了，并且，远程仓库的默认名称是origin。

要查看远程库的信息：git remote

推送分支，就是把该分支上的所有本地提交推送到远程库。推送时，要指定本地分支，这样，Git就会把该分支推送到远程库对应的远程分支上：git push origin master

从远程库clone时，默认情况下，在本地只能看到 master 分支，如果要抓取其他分支，需要在本地创建对应分支：git checkout -b dev origin/dev

然后关联起来：git branch --set-upstream-to=origin/dev dev

多人协作的工作模式通常是这样：

  1. 可以试图用 git push origin <branch-name> 推送自己的修改；

  2. 如果推送失败，则因为远程分支比你的本地更新，需要先用 git pull 试图合并；

    如果 git pull 提示 no tracking information，则说明本地分支和远程分支的链接关系没有创建，用命 令git branch --set-upstream-to=origin/ 建立连接

  3. 如果合并有冲突，则解决冲突，并在本地提交；

  4. 没有冲突或者解决掉冲突后，再用 git push origin <branch-name> 推送就能成功！

9. Rebase

发布一个版本时，我们通常先在版本库中打一个标签（tag），这样，就唯一确定了打标签时刻的版本。将来无论什么时候，取某个标签的版本，就是把那个打标签的时刻的历史版本取出来。所以，标签也是版本库的一个快照。

tag 就是一个让人容易记住的有意义的名字，它跟某个 commit 绑在一起。

创建标签：git tag v1.0

查看所有标签：git tag

为某个历史提交打标签：git tag v0.9 commit_id

查看标签信息：git show <tagname>

创建带有说明的标签，用 -a 指定标签名，-m 指定说明文字：git tag -a v0.1 -m "version 0.1 released” 1094adb

删除标签：git tag -d v0.1

推送某个标签到远程，使用命令：git push origin <tagname>

一次性推送全部尚未推送到远程的本地标签：git push origin —tags

如果标签已经推送到远程，要删除远程标签就麻烦一点:

  1. 先从本地删除：git tag -d v0.9

  2. 然后从远程删除：git push origin :refs/tags/v0.9

10. 版本回退

1. reset（不推荐）

通过reset的方式，把head指针指向之前的某次提交，reset之后，后面的版本就找不到了。

操作步骤如下：

  1. 在 github/gitlab上找到要恢复的版本号，如：

139dcfaa558e3276b30b6b2e5cbbb9c00bbdca96

一般前 8 位左右就足够定位一个版本了。

  1. 在客户端执行如下命令（执行前，先将本地代码切换到对应分支）：

git reset --hard 139dcfaa

  1. 强制push到对应的远程分支（如提交到develop分支）

git push -f -u origin develop

OK，现在到服务器上看到的代码就已经被还原回去了。这种操作存在一个问题，服务器上的代码虽然被还原了，但假如有多个人在使用，他们本地的版本依然是比服务器上的版本高的，所以，别人再重新提交代码的话，你撤销的操作又会被重新，你上面的操作也就白操作了。解决办法是，让别人把本地的分支先删掉，然后重新从服务器上拉取分支。

2. revert（推荐）

这种方式不会把版本往前回退，而是生成一个新的版本。所以，你只需要让别人更新一下代码就可以了，你之前操作的提交记录也会被保留下来。

操作步骤如下：

  1. 找到你误提交之前的版本号

  2. git revert -n 版本号

  3. git commit -m ‘’

  4. git push

OK，版本还原了，通知其他人更新一下代码吧

InnoDB

Posted on 2021-06-02 Edited on 2021-06-27 In 数据库 , MySQL
Symbols count in article: 371

在 InnoDB 中，表是根据主键顺序以索引的形式存放的，这种存储方式的表称为索引组织表。

每一个索引在 InnoDB 里面对应一棵 B+ 树。

索引类型包括主键索引和非主键索引：

  • 主键索引 - 也称聚簇索引，叶子节点存储的是整行数据。
  • 非主键索引 - 也称辅助索引、普通索引、二级索引，叶子节点存储的是主键的值。

基于主键索引的查询通过查询主键索引树可以直接得到整行数据；

而基于普通索引，得先查询普通索引树，得到主键的值，然后根据主键的值去查询主键索引树才能得到想要的数据，这个过称为回表。

显然，基于普通索引的查询比基于主键索引的查询需要多扫描一颗索引树，因此建议尽量通过主键查询。

MySQL索引

Posted on 2021-06-02 Edited on 2021-06-27 In 数据库 , MySQL
Symbols count in article: 3.2k

MySQL 官方对索引的定义为：索引（Index）是帮助 MySQL 高效获取数据的数据结构。

在 MySQL 中，索引是在存储引擎层实现的，因此不同的存储引擎可能有不一样的索引实现。

实现索引可以有多种方式，常用的有哈希表、B+树。

  • 哈希表

    哈希表是一种存储键值对的数据结构，根据 key 可以在 O(1) 时间内找到对应的 value。

    哈希表在等值（单个）查询时表现很好，但是因为哈希表不是有序的，因此在做区间查询时性能就很差。

  • B+树

    B+树的的所有叶节点是排序的，天然支持范围。
